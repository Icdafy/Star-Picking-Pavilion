# 摘星阁 v0.0.2 设计说明

**日期：** 2026-07-23  
**状态：** 已获用户确认  
**目标仓库：** `Icdafy/Star-Picking-Pavilion`  
**目标版本：** `v0.0.2`

## 1. 背景与目标

v0.0.2 解决两个已复现的桌面端问题：

1. 主题、常用网址星标等界面选择只在当前启动期间有效；完全退出后重新打开会恢复默认值。
2. DeepSeek API Key 提交后无法完成保存，运行时也无法使用该 Key 测试连接或分析新消息。

本版本在不降低随机回环端口、API 启动令牌、Electron sandbox 和 Windows 加密凭据保护的前提下，提供完整的界面选择持久化，并修复 DeepSeek 凭据确认链路。

## 2. 已确认根因

### 2.1 界面选择失忆

桌面应用每次启动都会为本地服务分配新的随机回环端口。Chromium `localStorage` 按完整 origin 隔离，端口变化会形成新的存储空间。现有实现因此只能通过同一端口下的页面刷新测试，无法跨应用重启恢复主题、网址星标和实时开关。

随机端口是本地 API 安全设计的一部分，本版本不改回固定端口。

### 2.2 DeepSeek 保存超时

后端 utility process 通过 `process.parentPort` 等待主进程返回凭据保存结果。Electron 在子进程侧触发的是 `MessageEvent`，业务消息位于 `event.data`；现有监听器却直接读取 `message.type`，因此忽略了主进程已经发回的成功确认。

实际结果是：

- Windows 加密凭据文件可能已经写入；
- 后端等待确认十秒后超时；
- 普通设置文件没有提交；
- 运行时 Key 没有更新；
- UI 最终只收到通用的内部错误，测试连接和后续 AI 分析均无法使用新 Key。

默认 DeepSeek 地址及 `deepseek-v4-flash`、`deepseek-v4-pro` 模型名与当前官方接口一致，不属于本次故障根因。

## 3. 方案比较与决策

### 方案 A：主进程界面偏好存储（采用）

在稳定的 Electron 用户数据目录中维护独立的 `ui-preferences.json`，由主进程负责读取、校验和原子写入，通过最小化 preload IPC 接口提供给渲染器。首屏主题从主进程内存同步读取，其余选择在应用初始化时恢复。

优点是跨随机端口稳定、首屏无主题闪烁、与采集配置及凭据事务解耦，并可集中进行字段白名单和版本迁移。缺点是需要新增一个小型持久化模块和 IPC 测试。

### 方案 B：合并到后端 `settings.json`

把界面状态作为后端设置字段保存。数据文件较少，但页面必须等待 HTTP API 后才能恢复，主题会出现闪烁，且界面偏好会与调度器、模型和凭据事务耦合，因此不采用。

### 方案 C：使用跨端口 Cookie

Cookie 不按端口隔离，改动量最小。但复杂集合需要额外编码，字段校验、版本升级、损坏恢复和非桌面运行边界都较弱，因此不作为长期方案。

## 4. 界面选择持久化

新增版本化偏好结构，只允许以下字段：

| 字段 | 允许值与默认值 |
| --- | --- |
| `theme` | `light` / `dark`，默认 `dark` |
| `view` | `featured` / `hot` / `all` / `daily` / `links` / `sources` / `settings`，默认 `featured` |
| `domain` | 空字符串 / `lowaltitude` / `aerospace`，默认空字符串 |
| `category` | 当前服务返回的分类文本；分类不存在时回退空字符串 |
| `dailyDate` | 有效的本地日历日期且不晚于今天；无效时回退当天 |
| `linksCategory` | 当前常用网址分类；分类不存在时回退“全部” |
| `commonLinksFavorites` | 去重后的已知网址 ID 数组；损坏时回退默认收藏 |
| `realtime` | 布尔值，默认 `true` |

“全部选择”指所有会改变界面模式、过滤范围或用户固定项的离散选择。搜索词、分页、滚动位置、卡片展开状态、未提交表单内容和提示消息属于临时操作，不持久化，避免旧查询或未完成输入在下次启动时意外恢复。

交互规则：

1. 主进程在创建窗口前加载并规范化偏好；文件不存在或损坏时使用安全默认值。
2. preload 只暴露“读取规范化偏好”和“提交允许字段补丁”两个能力，不暴露文件路径或任意文件访问。
3. `bootstrap.js` 在样式表加载前读取已缓存的主题，保证重启后首屏主题正确。
4. `app.js` 初始化状态后恢复最后页面与筛选；动态分类加载完成后再验证并应用分类。
5. 每次有效选择变化都提交最小补丁；写盘失败时保留当前会话状态并显示不包含路径或秘密的错误。
6. 主进程以临时文件加同目录原子替换写入，写入失败保留上一份有效文件。
7. 普通浏览器开发模式没有 preload 时继续使用当前 origin 的 `localStorage` 作为兼容回退。
8. 当前 origin 中可读取的旧主题、实时开关和星标会在首次运行时合并进入新偏好文件；无法访问的历史随机端口数据不尝试绕过 Chromium 隔离读取。

## 5. DeepSeek 凭据与调用链修复

- 子进程消息监听器从 Electron `MessageEvent.data` 解析 `credential:result`，同时保留对直接消息对象的兼容，便于普通 Node 测试和未来传输方式调整。
- 只有收到与 `requestId` 匹配的成功确认后，后端才更新内存中的 API Key 并继续提交非秘密设置。
- 主进程仍使用 Electron `safeStorage` 在 Windows 用户数据目录加密保存 API Key；`settings.json`、日志、HTTP 响应和测试输出不得包含明文 Key。
- 凭据保存或确认失败时，后端不宣称配置成功；UI 显示可操作的安全错误，并允许重试。
- 保存成功后，新 Key 无需重启即可用于“测试连接”和下一轮 AI 分析。重启后由主进程解密并注入后端运行时。
- 更换远程模型基础地址时仍要求同时提供新 Key，避免已有凭据被静默发送到不同服务。

## 6. 测试设计

### 6.1 单元测试

- 偏好文件不存在、字段损坏、未知字段、重复或未知网址 ID、无效日期和原子替换失败。
- 偏好补丁只接受允许字段和值，损坏数据恢复默认值。
- Electron `MessageEvent` 包装和普通直接消息都能完成对应 `requestId` 的凭据确认。
- 未匹配、重复和失败确认不会错误更新运行时 Key。

### 6.2 真实 Electron 回归

使用临时用户数据目录和本机回环模拟模型服务，连续启动两次真实 Electron：

1. 首次启动选择浅色主题、非默认页面、行业与分类、日报日期、网址分类、星标组合和关闭实时更新。
2. 保存测试 API Key 与模型设置，断言 UI 明确报告成功。
3. 点击测试连接，模拟服务验证收到了带认证的 `/chat/completions` 请求并返回合法 JSON。
4. 断言 `credentials.v1.json` 存在且不含明文 Key，`settings.json` 也不含 Key。
5. 完全关闭应用并使用同一用户数据目录重新启动。
6. 断言第二次启动使用不同随机端口，但所有界面选择、模型设置和“已配置 Key”状态均恢复。
7. 再次测试连接，确认重启后解密得到的 Key 能正常用于模型请求。

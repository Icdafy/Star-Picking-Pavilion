# Star-Picking-Pavilion v0.0.1 设计说明

**日期：** 2026-07-21  
**状态：** 已获用户确认  
**目标仓库：** `Icdafy/Star-Picking-Pavilion`  
**目标版本：** `v0.0.1`

## 1. 背景与目标

以现有“摘星阁”桌面应用为唯一主体，完整保留已经合并的“云幄 · 常用网址”功能和“云幄”名称，同时完成以下发布前收口：

- 中文产品名统一为“摘星阁”。
- 英文品牌统一为 `Star-Picking-Pavilion`；受 npm 命名规则限制时使用小写包名 `star-picking-pavilion`。
- 修复本地 API、窗口生命周期、键盘焦点、打包边界、依赖和数据迁移问题。
- 生成不包含源码工作树、数据库、日志、测试产物或密钥的 Windows 安装包。
- 为新仓库补齐 README、许可证、发布说明和可复现发布流程。
- 将最终代码合并到本地 `main`，推送至 `Icdafy/Star-Picking-Pavilion`，并发布首个 GitHub Release `v0.0.1`。

## 2. 方案比较与决策

### 方案 A：完整迁移并兼容旧数据（采用）

统一品牌标识，同时保留旧安装身份和已有用户数据的兼容迁移；对本地 API、打包流程和发布流水线做发布级加固。优点是风险最低、用户体验完整，缺点是需要增加一次性迁移逻辑和更多回归测试。

### 方案 B：全新应用、完全断开旧版本

全部改用新标识，不读取旧目录或旧数据库。实现最简单，但会让已经运行过“捕风司”或“摘星阁”旧构建的用户丢失本地配置和历史数据，因此不采用。

### 方案 C：只改名称并重新打包

仅替换名称、仓库地址和版本号。速度最快，但会保留本地 API 暴露、安装包污染、焦点丢失、依赖漏洞等发布阻断问题，因此不采用。

## 3. 品牌与产品身份

统一映射如下：

| 用途 | 值 |
| --- | --- |
| 中文产品名 / Windows 产品名 | `摘星阁` |
| 英文品牌 / GitHub 仓库 / 安装包前缀 | `Star-Picking-Pavilion` |
| npm 包名 | `star-picking-pavilion` |
| Electron appId | `com.icdafy.star-picking-pavilion` |
| preload 全局对象 | `window.starPickingPavilion` |

“云幄”只作为已合并功能的来源标签保留，例如“云幄 · 常用网址”；不得改名，也不引入云幄原项目的设置、密钥、发布物或其他功能。

安装器显式固定现有升级 GUID，使更换 appId 后仍能覆盖升级旧安装，而不是在 Windows 中并列安装第二份应用。代码中允许仅在兼容迁移模块和迁移测试里出现旧标识；面向用户的文案、当前配置和发布资料不得继续使用旧品牌。

## 4. 数据兼容与迁移

应用启动时在创建服务进程前运行幂等迁移：

1. 若新数据库已存在，不覆盖、不重复迁移。
2. 优先从当前 `%APPDATA%\摘星阁\windcatcher.db` 迁移到 `%APPDATA%\摘星阁\star-picking-pavilion.db`。
3. 当前目录不存在旧库时，再尝试 `%APPDATA%\捕风司\windcatcher.db`。
4. 使用 SQLite 一致性备份方式迁移，避免只复制主文件而遗漏 WAL 中尚未检查点的数据。
5. `settings.json` 仅在新目录没有相应配置时迁移；API 密钥继续只保存在本机用户数据目录，不进入仓库或安装包。
6. localStorage 首次读取新键时，可回退读取旧键并写入新键；迁移成功后使用新键，不删除旧键，确保可回滚。
7. 任一步失败都记录不含秘密值的错误并继续以安全默认配置启动，不破坏源数据。

仓库内 `data/` 和本机 `%APPDATA%` 数据均视为用户数据：不删除、不提交、不打包。

## 5. 本地服务安全

现有固定端口和 `Access-Control-Allow-Origin: *` 允许任意网页调用本地写接口，属于发布阻断。新架构如下：

- Electron 主进程在每次启动时生成高熵随机令牌，并选择可用的回环端口。
- 令牌和端口只通过子进程环境传递给本地服务，不写入磁盘、日志或渲染页面。
- Electron 会话只对自身发往该回环服务的 `/api/*` 请求注入认证请求头。
- 服务对所有状态变更接口以及包含设置的数据接口验证令牌；无效或缺失令牌返回 `403`。
- 开发模式若明确未配置令牌，仅允许同源请求；不返回通配 CORS，也不接受跨站凭据。
- 校验 `Host` 为回环地址和实际端口，限制请求体大小，异常请求返回明确的 `4xx`。
- 静态页面增加适合现有资源的 Content Security Policy 和基础安全响应头。
- API 响应、日志和错误消息不得回显 API 密钥或启动令牌。

此设计保留现有渲染器调用方式，不把秘密暴露给页面 JavaScript。

## 6. Windows 桌面运行时

- 使用 `app.requestSingleInstanceLock()` 保证单实例；重复启动时聚焦并恢复现有窗口。
- 本地服务启动失败、端口异常或意外退出时显示可理解的错误，而不是留下空白窗口。
- `BrowserWindow` 显式启用 `sandbox: true`，继续保持 `contextIsolation: true` 和 `nodeIntegration: false`。
- 外部打开仅允许 `http:` 和 `https:`，应用内部导航继续受限。
- preload 版本号从 Electron 当前应用版本读取，不硬编码。
- 自动更新只在打包版本启用，并指向新 GitHub 仓库。

## 7. 常用网址与可访问性

- 保持 14 条云幄常用网址、6 个分类、12 个默认收藏及现有排序和持久化语义不变。
- 测试夹具对 14 条完整对象做深度相等验证，防止名称、URL、描述、分类或默认收藏漂移。
- 分类切换和收藏切换重新渲染后恢复触发控件的键盘焦点；若控件已从当前视图消失，则落到对应区域的稳定焦点目标。
- 保持外链安全属性、响应式单列布局和无横向溢出行为。

发布审计发现其中包含共享文档、内部台账和明文 HTTP OA 入口。公开推送前设置强制人工门禁：用户必须在已获知风险后明确选择“原样公开 14 个网址”，或者选择“转为本地私有配置”。私有配置方案仍保留完整加载、分类、收藏和迁移能力，但公开仓库、Git 历史与公开安装包不得包含内部入口；未经明确选择不得推送或发布。

## 7.1 本地秘密与设置完整性

- 打包版本的 AI API Key 使用 Electron `safeStorage`（Windows DPAPI）加密后单独保存；明文只在主进程与服务进程运行内存中短暂存在。
- 旧 `settings.json` 中的明文 Key 首次启动时迁移到加密存储，验证成功后从普通设置中移除；迁移过程不在日志、错误或响应中回显 Key。
- 非秘密设置使用同目录临时文件和原子替换写入，写入失败保留上一份有效配置。
- AI 基础地址只接受 HTTPS；本机开发服务可例外使用 HTTP 回环地址。更换远程基础地址时要求重新输入 Key，防止已存凭据被发送到意外主机。
- 删除信源改为软停用，保证历史文章不会因关联信源物理删除而从界面消失。

## 8. 依赖与代码质量

- 升级存在已知漏洞或补丁版本落后的直接依赖，并通过锁文件固定结果。
- 若上游暂未提升传递依赖，则仅对已验证兼容的漏洞修复版本使用 npm `overrides`。
- 升级 `node-cron` 时用调度器行为测试覆盖启动、停止和表达式兼容。
- `npm audit --omit=dev` 必须达到无高危漏洞；剩余不可修复项如存在，需在发布说明中明确评估，否则不得发布。

## 10. 仓库与发布

新仓库已有独立初始提交。为避免强推和历史丢失：

1. 将现有旧远端保留为 `legacy`。
2. 将 `origin` 指向 `https://github.com/Icdafy/Star-Picking-Pavilion.git`。
3. 获取新仓库 `main`，用允许无共同祖先的普通合并纳入其初始提交，并以本项目完整 README 为准解决冲突。
4. 本地功能分支经验证后快进或普通合并回本地 `main`。
5. 推送 `main`，创建并推送签名需求之外的普通标签 `v0.0.1`。
6. GitHub Actions 从干净 Windows Runner 执行 `npm ci`、测试、生产依赖审计、打包、发布物验证和 SHA-256 生成，再发布 Release 资产。
7. 等待工作流完成并从 GitHub Release 下载资产复验哈希、文件名和可访问性。

首版暂按未代码签名安装包发布。README 和 Release Notes 必须明确 Windows SmartScreen 可能提示“未知发布者”，并给出核对 SHA-256 后选择“更多信息 → 仍要运行”的说明，不弱化风险。

README 至少包含：产品定位、功能、截图或界面说明、系统要求、安装步骤、SmartScreen 提示、数据位置与隐私、安全边界、开发/测试/构建命令、常用网址来源说明、许可证和 Release 链接。仓库补充 MIT `LICENSE`、`CHANGELOG.md` 与必要的安全说明。
